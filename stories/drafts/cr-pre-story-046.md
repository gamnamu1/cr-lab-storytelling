# Story 046: 코드로 쓰는 신뢰

**장르**: 사회 드라마
**주제**: 기술과 가치의 만남
**시점**: 3인칭 관찰자
**분량**: 약 8,400자
**주인공**: 시민M (38세, CR 기획자) / 이준호 (32세, 백엔드 개발자)
**시기**: 2024년 3월
**만남의 채널**: 온라인 개발자 커뮤니티

---

## 프롤로그

시민M은 새벽 2시에 온라인 개발자 커뮤니티에 글을 올렸다. 제목은 '언론 평가 플랫폼 개발 협력 요청'이었다. 글을 쓰기까지 3시간이 걸렸다. 어떻게 설명해야 이 아이디어가 단순한 망상이 아니라 실현 가능한 프로젝트로 들릴지 고민했다.

글의 핵심은 간단했다. 시민들이 언론 기사를 직접 평가하고, 그 평가가 축적되어 언론사의 신뢰도를 나타내는 시스템. 하지만 간단한 아이디어일수록 구현은 복잡했다. 데이터베이스 설계, API 구조, 프론트엔드 인터페이스, 보안, 확장성. 혼자서는 불가능했다.

글을 올리고 잠들었다. 아침 8시에 일어나 핸드폰을 확인했다. 조회수 237. 댓글 4개. 그 중 3개는 회의적이었다.

"또 언론 개혁 얘기? 이미 시도된 거 많은데 다 망했음"
"아이디어는 좋은데 수익 모델이 뭐임? 돈 안 되면 지속 불가능"
"시민 평가? 그거 결국 여론몰이 도구 되는 거 아님?"

하지만 네 번째 댓글이 달랐다.

"흥미롭네요. 기술적으로 구현 가능합니다. 다만 몇 가지 고민해야 할 지점이 있어요. DM 주시면 자세히 얘기 나눌 수 있습니다. - junholee_dev"

시민M은 즉시 DM을 보냈다.

## 1막: 첫 대화

이준호는 10년 차 백엔드 개발자였다. 스타트업 3곳을 거쳐 현재는 중견 IT기업에서 일했다. 주로 대규모 데이터 처리 시스템을 다뤘다. 일은 안정적이었고, 연봉도 괜찮았다. 하지만 의미는 없었다. 광고 플랫폼의 추천 알고리즘을 개선하고, 사용자 데이터를 분석하고, 전환율을 높이는 일. 기술적으로는 재미있었지만, 왜 하는지는 모르겠다는 생각을 자주 했다.

시민M의 DM을 받고 이준호는 회의실을 예약했다. 토요일 오후 2시, 강남역 근처 스터디 카페. 조용하고, 와이파이가 빠르고, 화이트보드가 있는 곳.

시민M이 먼저 도착해 있었다. 노트북을 열어놓고 뭔가를 정리하고 있었다. 이준호가 다가가자 고개를 들었다.

"이준호 님이시죠?"

"네. 시민M 님?"

"네. 앉으세요."

악수를 했다. 시민M의 손은 따뜻했다. 이준호는 자리에 앉았다.

"커뮤니티 글 잘 읽었어요. 언론 평가 플랫폼."

"네. 읽어주셔서 감사합니다."

시민M은 노트북을 돌려 화면을 보여줬다. 손으로 그린 시스템 아키텍처 다이어그램이었다. 사용자, 기사 DB, 평가 시스템, 신뢰도 산출 로직. 기본적인 구조는 갖춰져 있었다.

"이게 제가 생각한 구조예요. 근데 저는 개발자가 아니라서 이게 실제로 가능한지 잘 모르겠어요."

이준호는 다이어그램을 찬찬히 봤다. 아이디어 자체는 나쁘지 않았다. 하지만 세부 사항에서 문제가 보였다.

"일단 구현은 가능해요. 근데 몇 가지 질문이 있어요."

"말씀하세요."

"첫째, 기사를 어떻게 수집할 건가요? 언론사 RSS를 크롤링? 아니면 언론사 API?"

"크롤링을 생각했어요."

"그럼 저작권 문제가 있어요. 기사 전문을 가져오면 언론사가 고소할 수 있어요."

시민M의 표정이 어두워졌다. 생각하지 못한 부분이었다.

"그럼... 링크만 가져올 수 있나요?"

"그건 가능해요. 제목이랑 요약본 정도만 저장하고, 본문은 언론사 사이트로 연결하는 방식."

"그래도 평가는 할 수 있을까요?"

"할 수 있죠. 사용자가 기사를 읽고 돌아와서 평가하면 되니까."

이준호는 노트북을 꺼내 간단한 스케치를 그렸다. 사용자 플로우. 기사 목록 → 클릭 → 언론사 사이트로 이동 → 읽기 → 돌아와서 평가.

"둘째, 평가 기준이 뭐예요? 별점? 좋아요/싫어요? 아니면 항목별 평가?"

"항목별 평가를 생각했어요. 정확성, 공정성, 깊이, 이런 식으로."

"그럼 사용자가 각 항목을 이해할 수 있어야 해요. 정확성이 뭔지, 공정성이 뭔지. 설명이 필요하죠."

"맞아요. UI에 툴팁으로 설명을 넣으면 될까요?"

"그렇게 하면 돼요. 근데 문제는 사람들이 귀찮아한다는 거예요. 5개 항목 평가하려면 클릭을 5번 해야 하잖아요. 그냥 별점 하나 찍고 끝내고 싶어 할 거예요."

시민M은 고민에 잠겼다. 이준호의 말이 맞았다. 사용자 경험과 평가의 깊이 사이에서 균형을 잡아야 했다.

"그럼... 필수 항목은 전체 평가 하나만 받고, 세부 항목은 선택사항으로 하면 어떨까요?"

"그게 나을 것 같아요. 대부분은 전체 평가만 하고, 관심 있는 사람만 세부 항목 평가하는 거죠."

이준호는 계속 질문했다. 사용자 인증은 어떻게 할 것인지, 중복 평가는 어떻게 막을 것인지, 악의적 평가는 어떻게 걸러낼 것인지. 시민M은 생각한 부분은 대답하고, 생각하지 못한 부분은 메모했다.

한 시간이 지났다. 이준호의 노트는 질문과 스케치로 가득 찼다. 시민M의 노트도 마찬가지였다.

"마지막 질문이에요."

이준호가 말했다.

"왜 이걸 만들려고 하세요?"

시민M은 잠시 멈췄다. 커피를 한 모금 마셨다. 그리고 천천히 말했다.

"제가 언론을 못 믿겠어요. 같은 사건을 보도하는데 신문마다 완전히 다른 얘기를 해요. 누가 맞는지 알 수가 없어요. 그래서 뉴스 보는 걸 포기했어요. 몇 년 동안. 근데 그게 답은 아니잖아요. 시민이 뉴스를 안 보면 민주주의가 작동을 안 하니까. 그래서 생각했어요. 시민들이 직접 판단하면 어떨까. 각자 기사를 읽고 평가하고, 그 평가를 공유하면. 그럼 적어도 어느 언론이 더 신뢰할 만한지 알 수 있지 않을까."

이준호는 고개를 끄덕였다. 공감이 갔다. 그도 뉴스를 믿지 않았다. 댓글란을 보면 사람들이 서로 싸우고 있었다. "이게 사실이야", "아니야 저게 사실이야". 진실은 어디에도 없었다.

"이거 혼자 하시는 거예요?"

"지금은 네. 근데 혼자는 못 해요. 개발을 못하니까. 그래서 도움을 구하는 거예요."

"돈은 어떻게 할 건가요? 개발하려면 서버 비용도 들고, 시간도 들어요."

"처음엔 최소한으로 시작하려고 해요. AWS 프리티어 쓰고, 오픈소스 도구 활용하고. 제 시간은... 퇴근 후랑 주말을 쓸 생각이에요."

"그럼 저도 마찬가지겠네요. 퇴근 후랑 주말."

시민M이 이준호를 봤다. 놀란 표정이었다.

"도와주실 건가요?"

이준호는 대답하지 않았다. 아직 확신이 없었다.

## 2막: 기술과 가치

일주일 후, 이준호는 시민M에게 문서 하나를 보냈다. '언론 평가 플랫폼 기술 설계서'. 30페이지 분량. 시스템 아키텍처, 데이터 모델, API 명세, 보안 정책, 확장성 고려사항. 일주일 동안 퇴근 후 매일 3시간씩 작업한 결과였다.

왜 이렇게 공들여 썼을까. 이준호 자신도 잘 몰랐다. 시민M의 아이디어가 완벽하다고 생각하지 않았다. 성공할 거라고 확신하지도 못했다. 하지만 뭔가 끌렸다. 10년 동안 광고 플랫폼, 전자상거래, 게임 서버를 만들었다. 기술적으로는 재미있었지만 의미는 없었다. 이번에는 달랐다. 적어도 시도할 만한 가치가 있어 보였다.

시민M에게서 답이 왔다.

"정말 감사합니다. 이렇게까지 해주실 줄 몰랐어요. 내일 저녁 시간 있으세요? 밥 사고 싶어요."

다음 날 저녁, 두 사람은 강남역 근처 조용한 식당에서 만났다. 시민M은 30페이지 문서를 프린트해서 가져왔다. 형광펜으로 밑줄이 그어져 있고, 여백에 메모가 빼곡했다.

"다 읽으셨어요?"

"네. 세 번 읽었어요."

"질문 있으세요?"

"많아요."

밥을 먹으면서 시민M은 질문했다. 기술 용어를 이해하지 못하는 부분도 많았지만, 핵심은 파악하려고 애썼다. 이준호는 최대한 쉽게 설명했다.

"여기 '평가 신뢰도 가중치'라고 쓰셨는데, 이게 뭐예요?"

"모든 평가가 똑같은 무게를 가지면 안 돼요. 평소에 성실하게 평가하는 사람의 평가가 더 신뢰할 만하잖아요. 그래서 각 사용자의 평가 패턴을 분석해서 신뢰도를 계산하는 거예요."

"어떻게 계산하는데요?"

"여러 요소를 봐요. 평가 횟수, 평가의 일관성, 다른 사람들과의 합의 정도. 예를 들어, 누군가 매일 100개 기사를 1점으로 평가한다면, 이건 악의적이거나 봇일 가능성이 높아요. 그런 평가는 가중치를 낮춰야죠."

"그럼 반대로, 성실하게 평가하는 사람은?"

"가중치가 올라가요. 그 사람의 평가가 더 큰 영향을 미치는 거죠."

시민M은 고개를 끄덕였다. 합리적이었다.

"이거 만들려면 시간이 얼마나 걸릴까요?"

이준호는 계산했다. MVP(최소 기능 제품) 기준으로. 사용자 인증, 기사 크롤링, 평가 시스템, 신뢰도 계산, 프론트엔드. 주말마다 작업한다면.

"3개월 정도요. 매주 토요일 일요일 각각 8시간씩 작업한다고 가정하면."

"혼자서요?"

"네. 근데 프론트엔드는 제가 약해요. 디자인이랑 UX는 다른 사람 도움이 필요해요."

"제가 찾아볼게요. 디자이너 지인이 있어요."

식사가 끝나갈 무렵, 시민M이 조심스럽게 물었다.

"준호 님, 진짜로 같이 하실 건가요? 확답을 듣고 싶어요. 제가 다른 계획을 세워야 할지 알아야 해서요."

이준호는 잠시 생각했다. 3개월. 매주 16시간. 총 192시간. 회사 일도 있고, 개인 시간도 필요한데. 그걸 언론 평가 플랫폼에 쓴다고? 성공할지도 모르는 프로젝트에?

하지만 다른 생각도 들었다. 10년 동안 의미 없는 코드를 써왔다. 광고 클릭률을 0.1% 올리고, 로딩 속도를 0.2초 단축하고. 기술적으로는 성과였지만, 세상에 무슨 의미가 있었나. 이번에는 다르다. 적어도 시도할 만한 가치가 있다.

"할게요."

"정말요?"

"네. 근데 조건이 있어요."

"뭔데요?"

"이게 제 사이드 프로젝트라는 거 이해해주세요. 회사 일이 우선이에요. 데드라인을 못 지킬 수도 있어요."

"괜찮아요. 저도 제 일이 있으니까."

"그리고 중간에 안 될 것 같으면 접을 수 있어요. 서로 강제하지 말고."

"동의해요."

"마지막으로, 이게 성공하든 실패하든, 배우는 과정으로 생각할게요. 결과에 너무 집착하지 않을 거예요."

시민M은 잠시 생각하더니 손을 내밀었다.

"좋아요. 그렇게 해요. 가볍게 시작하는 거예요. 부담 없이."

악수를 했다. 이번에는 두 사람 모두의 손이 따뜻했다.

## 3막: 첫 코드

3월 둘째 주 토요일. 이준호는 카페에 노트북을 펼쳤다. GitHub에 새 repository를 만들었다. 이름: `citizen-review`. 설명: "A platform for citizens to evaluate news articles and media credibility."

첫 commit. `git commit -m "Initial commit"`. 빈 프로젝트. 하지만 시작이었다.

프로젝트 구조를 잡았다. Backend(Node.js + Express), Frontend(React), Database(PostgreSQL). 표준적인 스택. 이준호가 익숙한 도구들.

먼저 데이터 모델을 정의했다. Users, Articles, Evaluations, TrustScores. 테이블 간의 관계. Foreign keys. Indexes.

코드를 쓰기 시작했다. User authentication. JWT tokens. Password hashing. 기본적인 보안.

오후 3시쯤 시민M에게서 메시지가 왔다.

"작업 시작하셨어요?"

"네. 지금 사용자 인증 구현 중이에요."

"신기하다. 제가 상상만 하던 게 코드로 만들어지고 있네요."

"아직 아무것도 없어요. 로그인 기능뿐이에요."

"그래도 시작이잖아요."

이준호는 웃었다. 시민M의 열정이 느껴졌다.

다음 주말. Article crawler. RSS feeds를 읽어서 기사 메타데이터를 저장하는 스크립트. 제목, 요약, 링크, 발행 시간, 언론사. 10개 주요 언론사의 RSS를 등록했다.

스크립트를 실행했다. 콘솔에 로그가 흐르기 시작했다.

```
Fetching from 조선일보...
Saved 50 articles
Fetching from 한겨레...
Saved 48 articles
Fetching from 중앙일보...
Saved 52 articles
```

데이터베이스를 확인했다. Articles 테이블에 500개 기사가 저장돼 있었다. 실제 데이터. 이제 플랫폼이 실체를 갖기 시작했다.

시민M에게 스크린샷을 보냈다. 데이터베이스 쿼리 결과.

"와, 진짜 언론사 기사들이네요!"

"네. 매일 자동으로 수집하도록 cron job 설정할게요."

"언제쯤 프론트엔드 작업 시작해요?"

"다음 주부터요. 백엔드 API가 어느 정도 완성되면."

3월 셋째 주. Evaluation API. 사용자가 기사를 평가하는 엔드포인트. POST /api/evaluations. Request body: article_id, overall_score, accuracy_score (optional), fairness_score (optional).

코드를 쓰면서 이준호는 생각했다. 이 API를 통해 누군가의 평가가 저장된다. 그 평가가 쌓이면 언론사의 신뢰도가 계산된다. 그 신뢰도를 보고 다른 사람들이 판단한다. 단순한 CRUD(Create, Read, Update, Delete) 작업이지만, 의미는 컸다.

테스트 코드를 작성했다. API가 제대로 작동하는지 확인. 평가가 저장되는지, 중복 평가가 막히는지, 잘못된 입력이 거부되는지.

모든 테스트를 통과했다. 녹색 체크마크. 이준호는 만족스러웠다.

3월 넷째 주. Trust score calculation. 가장 복잡한 부분. 각 언론사의 평가를 집계하고, 가중 평균을 계산하고, 시간에 따라 감쇠시키고. 수식을 코드로 옮기는 작업.

주말 내내 매달렸다. 버그를 고치고, 엣지 케이스를 처리하고, 성능을 최적화하고.

일요일 밤 11시. 마지막 테스트를 돌렸다. 성공. 알고리즘이 작동했다. 조선일보 7.2점, 한겨레 7.5점, 중앙일보 6.8점. 물론 아직 실제 사용자 평가가 아니라 테스트 데이터였지만, 시스템은 작동했다.

시민M에게 메시지를 보냈다.

"신뢰도 계산 알고리즘 완성했어요."

답장이 빠르게 왔다.

"진짜요? 대박이에요!"

"다음 주부터 프론트엔드 작업 들어갈게요. 그럼 실제로 화면을 볼 수 있어요."

"너무 기대돼요. 벌써 3주 됐네요."

이준호는 달력을 봤다. 3주. 48시간. 그동안 이준호는 광고 플랫폼 대신 언론 평가 플랫폼을 만들었다. 회사에서는 추천 알고리즘을 개선했고, 집에서는 신뢰도 계산 알고리즘을 만들었다. 둘 다 코드였지만, 의미는 달랐다.

이준호는 노트북을 덮었다. 피곤했지만 기분이 좋았다. 오랜만에 느끼는 감정이었다. 의미 있는 일을 하고 있다는 느낌.
