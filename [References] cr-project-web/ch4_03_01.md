# **3-1. 프롬프트 템플릿 시스템** 
## **개요** 
CR 프로젝트는 기사의 품질을 정량적으로 측정하는 웹앱 플랫폼을 구축하는 것을 목표로 합니다. 이 프로젝트의 초기 단계에서는 '프롬프트 템플릿 방식'을 활용해 빠르게 기사 평가 시스템을 구현할 수 있습니다. 이는 미리 정의된 평가 기준과 지침이 담긴 템플릿을 ChatGPT나 Claude 같은 AI 모델에 제공하여 기사를 분석하도록 하는 방식입니다. 

이러한 접근법을 더욱 체계화하기 위해 'MCP(Model Context Protocol)' 서버/클라이언트 아키텍처를 도입하는 방안이 제안되었습니다. MCP는 간단히 말해 평가 기준과 알고리즘을 중앙에서 관리하고, 사용자들이 일관된 방식으로 이를 활용할 수 있게 해주는 통신 프로토콜입니다. 사용자는 복잡한 프롬프트를 직접 작성할 필요 없이 간단한 MCP 참조만으로 최신 평가 기준을 AI에 적용할 수 있습니다. 

더 나아가, MCP 아키텍처는 CR 프로젝트를 넘어 다양한 미디어 비평 플랫폼으로 확장될 수 있는 기반이 됩니다. 영상 뉴스 평가, 사설/칼럼 분석, 편향성 탐지 등 기획서 부록에 제시된 다양한 미디어 비평 도구들이 동일한 기술 인프라를 공유함으로써, 각자의 전문 영역에만 집중하면서도 하나의 강력한 미디어 비평 생태계를 구축할 수 있습니다. 
## **상세 설명** 
### **프롬프트 템플릿 시스템의 구현**
프롬프트 템플릿 시스템은 CR 프로젝트의 기획서에서 제시한 '투트랙 전략'의 Track 1(프롬프트 기반 분산형 시스템)에 해당합니다. 이 접근법은 기술적 진입 장벽이 낮아 빠르게 구현할 수 있고, 시민들의 참여를 통해 데이터를 수집할 수 있다는 장점이 있습니다. 

템플릿의 핵심 구성 요소는 다음과 같습니다: 

- 8개 핵심 평가 차원(진실성, 투명성, 균형성 등) 
- 30여 개의 문제적 패턴 목록(무주체 피동형, 익명 취재원 등) 
- 각 패턴이 평가 차원에 미치는 영향과 감점 기준 
- 기사 유형별 평가 방식 차별화 

초기에는 이러한 평가 기준을 GitHub 레포지토리에 호스팅하고, 웹검색이 가능한 AI 모델에게 이 레포지토리를 참조하여 기사를 평가하도록 할 수 있습니다. 이 방식은 프롬프트 길이 제한 문제를 해결하고, 평가 기준을 중앙에서 관리할 수 있다는 장점이 있습니다. 
### **MCP 서버/클라이언트 아키텍처**
MCP는 이러한 접근법을 더욱 체계화하고 확장하는 아키텍처입니다. 기본 작동 원리는 다음과 같습니다: 

1. 사용자가 프롬프트 요청을 MCP 서버에 전송 
2. 서버가 최신 평가 기준과 알고리즘을 적용해 최적화된 프롬프트 생성 
3. 최적화된 프롬프트가 사용자에게 전달되어 선택한 AI 모델에 입력 
4. 평가 결과 또한 MCP 서버에 저장되어 데이터베이스 구축 

예를 들어, 사용자는 다음과 같은 형식으로 AI에게 지시할 수 있습니다: 

다음 기사를 평가하기 위해 MCP://cr-project.org/evaluate?version=1.2&type=politics

평가 기준을 사용하여 분석해 주세요:

[기사 URL 또는 전문]

이 아키텍처의 주요 장점은: 

- 평가 기준의 중앙화: 모든 평가가 동일한 최신 기준으로 이루어짐 
- 데이터 수집 효율성: 평가 결과를 자동으로 수집하여 학습 데이터 확보 
- 사용자 경험 향상: 복잡한 프롬프트 작성 없이 간단한 참조만으로 평가 가능 
- 확장성: 기사 유형별, 주제별로 특화된 평가 기준을 쉽게 추가 가능 

기술적 구현 방안으로는 Node.js 또는 Python FastAPI 기반의 경량 RESTful API 서버와 평가 기준 데이터베이스를 구축하고, 브라우저 확장 프로그램이나 웹 인터페이스를 클라이언트로 제공할 수 있습니다. 
### **미디어 비평 생태계로의 확장**
MCP 아키텍처의 가장 큰 가치는 CR 프로젝트를 넘어 다양한 미디어 비평 플랫폼을 위한 공통 인프라로 활용될 수 있다는 점입니다. 기획서 부록에 제시된 다양한 비평 플랫폼들이 각각 독립적으로 개발되는 대신, MCP라는 공통 프로토콜을 공유함으로써: 

- 각 팀은 자신의 전문 영역(평가 기준 정의)에만 집중 가능 
- 기술 인프라는 공유함으로써 중복 개발 방지 
- 네트워크 효과: 각 플랫폼의 발전이 전체 생태계를 강화 

예를 들어, 다음과 같은 특화 모듈들이 동일한 MCP 아키텍처 위에서 개발될 수 있습니다: 

- 영상 뉴스 평가: MCP://protocol/video?format=youtube 
- 사설/칼럼 평가: MCP://protocol/editorial 
- 편향성 분석: MCP://protocol/analyze/bias 
### **프로젝트 로드맵에서의 위치**
프롬프트 템플릿과 MCP 아키텍처는 CR 프로젝트의 단계적 구현 전략에서 다음과 같은 위치를 가집니다: 

1. **초기 단계(0-1개월)** : 기본 프롬프트 템플릿 개발 및 GitHub 레포지토리 구축 
2. **개발 단계(1-3개월)** : MCP 프로토콜 정의 및 기본 서버 구축 
3. **확장 단계(3-6개월)** : 다양한 클라이언트 개발 및 특화 모듈 추가 
4. **통합 단계(6개월 이후)** : 수집된 데이터를 활용한 맞춤형 AI 모델 훈련(Track 2) 

이 접근법은 기획서에서 강조한 "접근성, 개방성, 확장성"이라는 핵심 원칙과 "비영리를 철저히 고수"한다는 가치에 부합하면서도, 기술적으로 실현 가능한 솔루션을 제공합니다. 궁극적으로는 "시민 주도의 언론 개혁"이라는 CR 프로젝트의 비전을 더 빠르고 효과적으로 실현하는 데 기여할 것입니다.

\### 평가 차원 (8개)

\#### 1. 진실성과 정확성

사실 검증 부실, 이차 자료 의존, 오보 관리 부실, 사실과 의견 혼재, 데이터/통계 오용

\#### 2. 투명성과 책임성

취재원 투명성 결여, 책임 회피 표현, 기자/매체 정체성 불투명

\#### 3. 균형성과 공정성

관점 다양성 부족, 편향적 보도, 반론권 무시

\#### 4. 독립성과 자율성

권력 유착, 상업적 이익 우선, 경쟁 중시 보도, 자기검열

\#### 5. 인권과 프라이버시 존중

불법/비윤리적 취재, 개인정보 보호 위반, 명예훼손과 인권침해, 사법 절차 위반

\#### 6. 전문성과 심층성

맥락 부재, 전문성 부족, 단순 정보 전달, 심층 분석 부재

\#### 7. 언어와 표현의 윤리

과장/왜곡 표현, 선정적 표현, 비속어와 혐오 표현, 잘못된 외래어/약어 사용

\#### 8. 디지털 환경의 윤리

클릭베이트, 속보성 오류, 디지털 윤리 위반, 알고리즘 편향성

\- \*\*문제적 패턴: [패턴명]\*\* (굵은 글씨로 표시)

\- 문제점에 대한 구체적 설명 (기사 원문 인용 포함, 2-3 문단)

\- \*\*윤리 위반 근거\*\*: (굵은 글씨로 표시)

`  `- 관련 윤리 규범을 정확히 인용 (불릿 포인트 사용)

`  `- 예: "언론윤리헌장 제1조: "진실을 추구한다 - 모든 정보를 성실하게 검증하고 명확한 근거를 바탕으로 보도한다.""

\## 문제적 보도 관행 분류

\### 1. 진실성과 정확성 관련

\#### 1-1. 사실 검증 부실

\- \*\*검증 절차 생략\*\*: 정보의 정확성을 충분히 확인하지 않고 보도

\- \*\*크로스 체크 부재\*\*: 취재 사실에 대한 충분한 교차 검증 미실시

\- \*\*투 소스 룰 무시\*\*: 중요 사안에 대해 두 개 이상의 독립적 출처 확인 생략

\- \*\*근거 자료 미제시\*\*: 출처 불명확한 인용, 통계나 연구 발표 인용 시 구체적 근거 미제시

\- \*\*커뮤니티 저널리즘\*\*: 인터넷 커뮤니티나 SNS 글을 추가 확인 없이 그대로 기사화

\- \*\*취재원 발언의 사실관계 확인 부재\*\*

\- \*\*사법 절차 존중 위반 보도\*\*: 수사기관 제공 피의 사실 추가 확인 없이 보도

\#### 1-2. 이차 자료 의존

\- \*\*보도자료 저널리즘\*\*: 보도자료를 검증 없이 그대로 보도

\- \*\*받아쓰기 저널리즘\*\*: 해외 언론 기사를 그대로 받아쓰거나 여러 언론사 기사를 짜깁기

\- \*\*지라시 의존\*\*: 확인되지 않은 정보나 소문에 의존

\- \*\*인용 출처 미표기\*\*: 다른 매체의 보도를 인용하면서 출처를 밝히지 않음

\- \*\*연합뉴스 저널리즘\*\*: 연합뉴스 기사를 사실관계 확인 없이 자사 뉴스로 보도

\- \*\*특파원 저널리즘\*\*: 외신을 단순 번역하여 특파원 바이라인으로 보도

\#### 1-3. 오보 관리 부실

\- \*\*오보에 대한 소극적 정정\*\*: 오보를 내고도 책임지지 않거나 제대로 정정하지 않음

\- \*\*반복적 오보 방치\*\*: 동일한 오류를 반복하는 문제

# **3-3. 기사 정보 수집 기술 가이드** 
### **개요** 
이 부록은 CR(Checking the Reliability of news) 플랫폼을 실제로 구현하는 기술팀과 프로젝트에 관심 있는 시민 전문가들을 위한 참고 자료입니다. 기사의 품질을 정량적으로 측정하는 이 프로젝트에서 가장 기본이 되는 '기사 정보를 어떻게 수집하고 처리할 것인가'에 대한 기술적 접근법을 담고 있습니다. 

뉴스 기사를 분석하기 위해서는 먼저 기사 콘텐츠를 정확하게 가져와야 합니다. 이는 마치 의사가 진료를 하기 전에 환자의 상태를 정확히 파악하는 것과 같습니다. 인터넷에 게시된 기사들은 언론사마다 다른 형식(HTML 구조)을 가지고 있으며, 본문 외에도 광고, 관련 기사 링크, 구독 유도 문구 등 분석에 불필요한 요소들이 함께 포함되어 있습니다. 또한 일부 언론사는 기사 복사를 방지하는 기술적 장치를 사용하기도 합니다. 

이 부록에서는 다음과 같은 내용을 다룹니다: 

- 다양한 언론사 웹사이트에서 기사 본문만을 정확히 추출하는 방법 
- 복사 방지 기능이 있는 웹사이트에서도 합법적으로 텍스트를 획득하는 방법 
- 모바일과 데스크톱 등 다양한 환경에서 일관된 결과를 얻는 기술 
- 실제 구현에 활용할 수 있는 코드 예시와 라이브러리 소개 
- 오류 처리와 성능 최적화 방안 

기술적인 배경이 없는 시민 참여자들도 이 내용을 통해 CR 프로젝트가 어떤 기술적 기반 위에서 작동하는지 이해할 수 있을 것입니다. 또한 개발 경험이 있는 참여자라면 직접 테스트해보거나 개선 아이디어를 제안하는 데 참고할 수 있습니다.

## **1. 기사 콘텐츠 추출 기술** 
### **1.1 HTML 구조 분석 및 본문 필터링**

- **머신러닝 기반 필터링** : 본문/비본문 요소 분류 모델 적용 
- **DOM 트리 분석** : 주요 언론사별 HTML 구조 패턴 매핑 
### **1.2 본문 정제 파이프라인**

### **2.1 합법적 텍스트 획득 방법**
- **브라우저 읽기 모드** : 주요 브라우저의 Reader Mode 활용 
- **OCR 적용** : 스크린샷을 통한 텍스트 인식 (Tesseract, Google Vision API) 
- **인쇄 미리보기** : 인쇄용 페이지를 통한 텍스트 추출 
### **2.2 법적 고려사항**
- 공개된 기사에 대한 개인적 연구/비영리 사용 범위 준수 
- 저작권법의 공정 이용(fair use) 원칙 적용 
- 유료 콘텐츠 우회 시도 배제 
## **3. 디바이스 및 브라우저 최적화**

# **3-4. 사용자 경험 시나리오** 
## **구체적 사용자 경험 시나리오 (URL 입력 기준)** 
1. 사용자가 웹앱에 접속하여 메인 페이지에서 "URL로 분석하기" 선택 
2. 분석할 뉴스 기사의 URL을 입력 필드에 붙여넣고 "분석 시작" 버튼 클릭 
3. 시스템이 로딩 애니메이션과 함께 다음 단계 진행: 
   1. "URL 확인 중..." (3초) 
   1. "기사 내용 추출 중..." (5초) 
   1. "기사 유형 분석 중..." (3초) 
   1. "품질 평가 진행 중..." (10초) 
4. 기사가 분석 가능한 유형(정치/사회 분야 스트레이트 뉴스)으로 확인되면: 
   1. 화면 상단에 종합 품질 점수(78/100점) 표시 
   1. 8개 평가 차원을 레이더 차트로 시각화 
   1. 기사 원문 표시 (문제 표현에 하이라이트 적용) 
5. 사용자가 하이라이트된 문장에 마우스 오버하면: 
   1. 툴팁으로 문제점 상세 설명 표시 
   1. 예: "무주체 피동형 표현('알려졌다')은 정보 출처를 불명확하게 만들어 투명성과 진실성을 저해합니다." 
   1. 개선 제안: "~회사 대변인이 밝혔다" 또는 "~보도자료에 따르면" 등으로 출처를 명확히 하는 것이 좋습니다. 
6. 상세 분석 탭에서: 
   1. 각 평가 차원별 세부 점수 및 근거 확인 
   1. 문제 패턴 발견 빈도 및 심각도 확인 
   1. 주요 개선 포인트 5가지 제시 
7. "PDF 리포트 다운로드" 버튼 클릭 시: 
   1. 분석 결과가 포함된 PDF 파일 생성 및 다운로드 
   1. 기사 원문, 평가 점수, 개선 제안이 포함된 전문 보고서 
8. "결과 공유하기" 버튼 클릭 시: 
   1. 고유 URL이 생성되어 클립보드에 복사 
   1. 소셜 미디어 공유 옵션 표시 
9. "커뮤니티 토론" 버튼 클릭 시: 
   1. Discord 서버 연결 및 해당 분석 결과 전용 채널로 이동 
   1. 다른 사용자들과 해당 기사의 품질에 대해 토론 가능 
10. 분석 완료 후: 
    1. "다른 기사 분석하기" 버튼으로 새로운 분석 시작 가능 
    1. 로그인한 사용자는 "분석 히스토리"에서 이전 분석 결과 확인 가능 

이 최종 서비스 흐름도는 포괄적인 기능성과 사용자 경험을 기반으로, 에러 처리와 예외 상황 관리를 체계적으로 통합하여 더욱 안정적이고 사용자 친화적인 시스템을 구현합니다.
